# daed 严格订阅同步设计

## 背景

#37 包含两个独立问题：

1. daed 原生面板更新订阅后只改数据库，没有自动应用运行态；上次修复只覆盖 LuCI 设置页按钮和 cron，未覆盖原生面板按钮。
2. dae-wing 会保留被策略组单独引用的旧订阅节点。用户逐个选节点后，订阅更新可能让旧节点持续累积。

目标是让 daed 原生面板、LuCI 手动更新和自动更新得到一致结果，同时不误删用户明确固定的节点。

## 行为规则

- 更新订阅时以本次拉取并解析出的节点为准，删除已经不在订阅中的旧节点。
- 仍被 `fixed` 策略组引用的失效旧节点不删除，而是转成独立节点继续保留。
- 仍在新订阅中的节点保持原 ID 和策略组关系，避免无意义地删除再创建。
- 普通策略组里引用的失效旧节点随订阅一起删除。
- 更新全部成功后立即应用运行态；应用失败必须返回错误并写入日志。
- 保持 `updateSubscription` 的 GraphQL 签名不变，在本项目的 daed 补丁中改为严格同步并在运行中自动应用。这样无需重建已经嵌入二进制的原生 Web，也能覆盖原生按钮和 daed 自带 cron。

## 方案比较

### 方案 A：补强现有更新接口（采用）

在 dae-wing 的现有 `updateSubscription` 调用链中完成新旧节点对比、固定节点保护、失效节点清理、新节点导入和运行态应用。GraphQL 签名不变，因此 daed 原生面板、daed 自带 cron 及 LuCI 更新脚本会自动得到一致行为。

优点是判断准确、失败可整体回滚，只需维护一份小型 dae-wing 补丁。缺点是本项目里的 `updateSubscription` 语义会比上游更严格，升级 dae-wing 时需要验证补丁。

### 方案 B：直接改变原接口语义

直接让 `updateSubscription` 默认删除失效节点。改动表面更少，但会影响所有调用者，也偏离上游明确的保留规则，升级时风险更高，因此不采用。

### 方案 C：Shell 调多个 GraphQL 接口清理

更新前后由 Shell 逐个解绑、删除和重绑节点。该流程不是原子操作，且现有接口无法可靠判断哪些节点属于本次新订阅，失败时可能留下空策略组，因此不采用。

## 数据流

1. 调用订阅更新接口并拉取最新订阅内容。
2. 解析并规范化新节点链接，建立本次订阅节点集合。
3. 查询原订阅节点及其策略组引用。
4. 新旧集合中都存在的节点原地保留。
5. 只存在于旧集合的节点：
   - 被任一 `fixed` 组引用：清除订阅归属，作为独立节点保留；
   - 只被普通组引用或未被引用：删除节点及相关组引用。
6. 导入只存在于新集合的节点。
7. 检查事务结束后订阅仍有有效节点，再提交事务。
8. 数据库提交成功后，如果 daed 当前正在运行代理，则排队并合并运行态应用；失败时保留数据库更新和 modified 状态，并向调用方明确报错，方便重试。

## 调用入口

- daed 原生面板：现有更新按钮继续调用 `updateSubscription`，后端完成严格同步和运行态应用。
- LuCI 设置页：`daed-sub-update.sh` 继续调用同一个接口；脚本末尾保留一次统一应用，兼容尚未升级 daed 的组合安装。
- daed 自带 cron：继续调用同一后端更新函数，自动严格同步并应用运行态。
- LuCI cron：继续调用 `daed-sub-update.sh`，行为与 LuCI 手动更新一致。

## 错误处理

- 下载、解析或数据库处理失败时回滚该订阅更新，不留下半更新状态。
- 任一订阅失败时不显示整体成功，也不应用不完整的运行态。
- 运行态应用失败时保留已经成功提交的数据库内容和待应用标记，但返回明确错误，方便用户再次应用或重试。
- 日志不得记录订阅地址、节点链接、密码或令牌。

## 验证标准

- 未被策略组使用的失效旧节点会删除。
- 普通策略组中的失效旧节点会从组和订阅中删除。
- `fixed` 组中的失效旧节点会转为独立节点，节点 ID 和组关系不变。
- 仍在订阅中的旧节点保持 ID 和组关系。
- 整个订阅绑定到策略组时，节点随订阅正常增删。
- 新旧订阅内容完全相同时更新成功，不出现“没有可导入节点”的误报。
- 更新失败时事务回滚。
- daed 原生面板、LuCI 手动按钮和 cron 三条入口结果一致。
- 更新后运行态包含新节点，不再需要用户手动重启。
- 252 上完成真实订阅更新复测，并检查更新日志、节点数量、策略组关系和运行态。
